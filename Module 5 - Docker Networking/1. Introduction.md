# Docker Networking

**Docker Networking**

**Docker Networking** is the system that allows containers to:
- Talk to **each other**
- Talk to the **host machine**
- Talk to the **outside world (internet)**
- Be **completely isolated** from everything

- By default, every container gets its own **network namespace** — its own
- isolated network stack with its own IP address, routing table, and ports.

---

## Why Docker Networking?

**Docker Networking — Problem & Solution**

**Without Docker Networking**

- Containers are completely isolated by default — they cannot talk to each other
- There is no way to expose your containerized service to the internet or outside world
- There is no isolation between apps — all processes share the same network, causing conflicts
- Containers running on different servers cannot communicate with each other at all

Without networking — your containers are like computers with no network cable plugged in


**With Docker Networking**

- An App container can talk to a DB container directly using just the container name — no IP address needed
- A Web server container can be exposed on port 80 — accessible from the internet via port mapping
- Internal services like databases are hidden from the outside world — only containers on the same network can reach them
- Containers running on completely different servers can communicate with each other using Overlay networking


With networking — your containers work together as one connected system

---

**The Golden Rules**

- Always use custom bridge — not the default bridge — to get DNS by container name
- Use port mapping (-p) to expose a container port to the outside world
- Keep databases on a private network — never expose them directly to the internet
- Use overlay network when containers need to talk across different servers
- One container can be connected to multiple networks at the same time

## Docker Network Drivers

**Network Drivers**

- Docker Network Drivers are the different types of networking modes that Docker provides — each one controls how containers communicate with each other, with the host machine, and with the outside world. 

- Think of network drivers like different types of network cables — each one connects things differently depending on what you need.

- Docker provides **5 built-in network drivers**:

---

- The first and most common is the **Bridge driver**.

  - This is the default.
  - When you run any container without specifying a network, it automatically joins a virtual bridge network called docker0.
  - Containers on the same bridge can talk to each other using IP addresses, and they can reach the internet through NAT.
  - But the important thing to know is that on the default bridge, containers cannot reach each other by name — only by IP.
  - That is why we always create a Custom Bridge Network in production — because on a custom bridge, Docker runs a built-in DNS server that lets containers talk to each other simply by container name.
  - So instead of connecting to 172.17.0.3, your app container can just say 'connect to database' — and Docker resolves it automatically.

- The second driver is **Host.**

  - When you use the host network, the container shares the host machine's network stack directly — no isolation, no NAT, no port mapping needed.
  - The container uses the host's IP address and binds directly to host ports.
  - This gives the best network performance because there is no overhead, but it sacrifices network isolation.
  - It is used for high-performance applications or network monitoring tools.
    
- The third driver is **None.**
  
  - This completely disables networking for the container.
  - The container gets no network interface, no IP address — it cannot send or receive any network traffic.
  - It is used for maximum security isolation or for batch processing jobs that simply do not need any network access.
 
- The fourth driver is **Overlay.**

  - This is what makes multi-host networking possible.
  - When you have containers running on different physical servers — like in a Docker Swarm cluster — overlay networks let them communicate as if they were on the same local network.
  - Under the hood it uses VXLAN tunneling to encapsulate container traffic and send it over the physical network between hosts.
  - Kubernetes uses a similar concept with its own CNI plugins like Calico and Flannel.

- The fifth driver is **Macvlan.**

  - This assigns a real MAC address to a container — making it appear as a physical device directly on your network with its own IP address on the LAN.
  - It is mainly used for legacy applications that need to be on the physical network directly, or for network monitoring tools that need a real MAC address."

**Summary**

---

| Driver | Scope | Use Case |
|--------|-------|----------|
| **bridge** | Single host | Default, most containers |
| **host** | Single host | High performance, no isolation |
| **none** | Single host | Maximum isolation, no network |
| **overlay** | Multi-host | Docker Swarm, distributed apps |
| **macvlan** | Single host | Legacy apps needing direct network |

## Custom Bridge Network

Always use a **custom bridge network** instead of the default bridge — you get:

- DNS name resolution between containers
- Better isolation
- Dynamic connect/disconnect
- Custom subnet and gateway

```bash
# Create a custom bridge network
docker network create mynetwork

# Create with custom subnet
docker network create \
  --driver bridge \
  --subnet 192.168.100.0/24 \
  --gateway 192.168.100.1 \
  mynetwork

# Run containers on custom network
docker run -d --name webapp --network mynetwork nginx
docker run -d --name database --network mynetwork postgres

# Now containers can talk by NAME! 
docker exec webapp ping database
# PING database (192.168.100.3): 56 data bytes
# 64 bytes from 192.168.100.3: seq=0 ttl=64
```

**Connect a Running Container to a Network**

```bash
# Add container to a network
docker network connect mynetwork mycontainer

# Remove container from a network
docker network disconnect mynetwork mycontainer

# A container can be on MULTIPLE networks!
docker network connect network1 mycontainer
docker network connect network2 mycontainer
```

## Container DNS & Name Resolution

On a **custom bridge network**, Docker runs a built-in **DNS server** at `127.0.0.11`.

```
Container (webapp) wants to reach "database"
           │
           ▼
   Queries Docker DNS: 127.0.0.11
           │
           ▼
   DNS resolves "database" → 192.168.100.3
           │
           ▼
   webapp connects to database 
```

```bash
# Check DNS inside a container
docker exec webapp cat /etc/resolv.conf
# nameserver 127.0.0.11   ← Docker's built-in DNS
# options ndots:0

# Test DNS resolution
docker exec webapp nslookup database
# Server:    127.0.0.11
# Address:   127.0.0.11:53
# Name:      database
# Address:   192.168.100.3
```

**DNS Rules:**

- Custom bridge  → DNS by container name works
- Overlay        → DNS by service name works
- Default bridge → DNS by name does NOT work (use IP)
- Host network   → No Docker DNS (uses host DNS)

## Port Mapping & Publishing

Port mapping connects a **host port** to a **container port**.

```
User Browser                Docker Host                Container
     │                           │                         │
     │  http://host:8080         │                         │
     │──────────────────────────►│                         │
     │                           │  forward to port 80     │
     │                           │────────────────────────►│
     │                           │                         │ (nginx listens on 80)
     │                           │◄────────────────────────│
     │◄──────────────────────────│                         │
```

```bash
# Map host port 8080 to container port 80
docker run -d -p 8080:80 nginx

# Map multiple ports
docker run -d -p 8080:80 -p 443:443 nginx

# Bind to specific host IP
docker run -d -p 127.0.0.1:8080:80 nginx   # localhost only
docker run -d -p 0.0.0.0:8080:80 nginx     # all interfaces (default)

# Random host port (Docker picks available port)
docker run -d -p 80 nginx

# See which port was assigned
docker port mycontainer

# UDP port mapping
docker run -d -p 5353:5353/udp mydns
```

**Port Mapping Table:**

| Syntax | Meaning |
|--------|---------|
| `-p 8080:80` | Host 8080 → Container 80 |
| `-p 80` | Random host port → Container 80 |
| `-p 127.0.0.1:8080:80` | Localhost only → Container 80 |
| `-p 8080:80/tcp` | TCP only |
| `-p 5353:5353/udp` | UDP mapping |
