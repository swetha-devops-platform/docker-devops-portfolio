# Docker Bind Mounts & Volumes

# Problems with Containers

**1) Data Loss Problem**

For example, Nginx is deployed inside a container and stores information such as:

* Who logged in
* User details
* Log files

If the container crashes, stops, or gets deleted, all the data inside the container will also be deleted.

Containers are **ephemeral in nature**, which means data is temporary and not permanently stored.

---

**2) Classic Frontend & Backend File Sharing Problem**

```text id="4jlwm7"
Frontend  <---- Files ---->  Backend
```

* Frontend may upload files (example: YAML, JSON, HTML files)
* Backend processes those files
* If backend container goes down, files generated or processed by it may also disappear
* This can lead to data loss and application issues

---

**3) Application Container File Access Problem**

```text id="k5jlwm"
Application Container  →  Files
```

The application inside the container needs to:

* Read files
* Store files
* Display files to users

If files are stored only inside the container, they may be lost when the container is removed.

To solve these problems, Docker provides three concepts:

**1) Bind Mounts**

---

- A Bind Mount maps a specific folder on the host directly into the container
- You choose exactly which host path to mount
- Changes made inside the container are immediately reflected on the host
- Changes made on the host are immediately reflected inside the container
- Perfect for local development — edit code on host, see changes in container instantly
- The host path must exist before mounting

---

**2) Volumes**

- A Docker Volume is a dedicated storage area managed completely by Docker
- Stored on the host at /var/lib/docker/volumes/
- Completely independent of the container lifecycle
- Data survives container stop, restart, and removal
- Can be shared between multiple containers
- Can be backed up, restored, and migrated easily

```
/var/lib/docker/
└── volumes/
    ├── mydata/
    │   └── _data/         ← your actual data lives here
    ├── dbdata/
    │   └── _data/
    └── logdata/
        └── _data/
```
---

**3. tmpfs Mount**

- A tmpfs mount stores data in the host machine's memory (RAM)
- Data is never written to disk
- Data is lost when the container stops
- Perfect for sensitive data that should not persist
- Much faster than disk-based storage

---


# Difference Between Bind Mounts and Volumes

| Bind Mounts                 | Volumes                           |
| --------------------------- | --------------------------------- |
| Uses host machine directory | Managed by Docker                 |
| Depends on host file path   | Independent of host path          |
| Less portable               | More portable                     |
| Mostly used in development  | Mostly used in production         |
| Manual management           | Docker manages lifecycle          |
| Direct access to host files | Stored inside Docker storage area |
