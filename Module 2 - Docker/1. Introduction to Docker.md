# Introduction to Docker

---

What is Docker?

- **Docker** is an open-source **containerization platform** that allows you to package an application along with all its dependencies — code, runtime, libraries, and configuration — into a single portable unit called a **container**.

- **Core Idea:** *"Build once, run anywhere"* — eliminating the classic **"it works on my machine"** problem.

---

## Why Docker?

---
Docker is very popular because it simplifies application deployment and ensures consistency across different environments.

The main reasons for Docker’s popularity are:

**1. Portability** - Docker containers can run consistently across development, testing, staging, and production environments without compatibility issues.

**2. Lightweight Architecture** - Containers share the host operating system kernel, making them faster and more resource-efficient than virtual machines.

**3. Faster Deployment** - Containers start within seconds, which improves application deployment speed and reduces downtime.

**4. Scalability** - Docker integrates easily with orchestration tools like Kubernetes, making it suitable for scaling modern applications.

**5. DevOps and CI/CD Support** - Docker works efficiently with CI/CD pipelines, enabling automated builds, testing, and deployments.

**6. Application Isolation** - Each container runs in an isolated environment, reducing dependency conflicts between applications.

**7. Cloud Compatibility** - Docker is widely supported in cloud platforms like AWS, Azure, and Google Cloud.

Because of these advantages, Docker has become a standard containerization platform in modern DevOps and cloud-native application development.”

# Docker Architecture

---

---

- Docker follows a Client-Server architecture 

- At the very top is the Docker Client. This is what the developer interacts with — the CLI. When you type any command like docker build or docker run or docker pull, you are talking to the Docker Client. The client itself does not do any work. It takes your command, converts it into a REST API call, and sends it over a Unix socket to the Docker Daemon. Think of the client as the waiter — it takes your order and passes it to the kitchen.

- The Docker Daemon — called dockerd — is the kitchen. It is a background service running on the host machine, listening for API requests. The moment it receives a request, it acts on it. It is responsible for managing everything — building images, creating containers, setting up networks, and managing volumes. The daemon is the brain of the entire Docker system.

- Now, the daemon does not do everything alone. Internally it delegates container management to containerd. containerd is an industry-standard container runtime — it is actually a CNCF graduated project used directly by Kubernetes as well. containerd handles the container lifecycle — pulling images from the registry, managing image layers, creating container snapshots, and preparing the container to run.

- But even containerd does not create the container itself. It hands off to runc — the lowest level component. runc is an OCI-compliant runtime that directly talks to the Linux kernel. It creates the namespaces for isolation — process isolation, network isolation, filesystem isolation — and sets up cgroups to limit how much CPU and memory the container can use. Once the container process is started, runc exits. Its job is done. containerd then monitors the running container.

- On the storage side, Docker uses a layered filesystem called OverlayFS. Every Docker Image is made up of read-only layers — one layer per Dockerfile instruction. When you run a container, Docker adds a thin writable layer on top of those read-only image layers. Any changes the container makes go into that writable layer. When the container is deleted, that layer is gone. The image layers underneath remain untouched and are shared across all containers running from the same image.

- On the networking side, Docker creates a virtual bridge network called docker0 on the host. Every container gets connected to this bridge and gets its own IP address. Port mapping connects a host port to a container port — that is how traffic from the outside world reaches your container.

- And finally — the Docker Registry. When the daemon needs an image that is not available locally, it pulls it from a registry. Docker Hub is the default public registry. In production, companies use private registries like AWS ECR, Google GCR, or Azure ACR. Once pulled, the image is cached locally so the daemon does not need to pull it again.

- And if you are on a developer machine — Mac or Windows — all of this is bundled inside Docker Desktop, which gives you the daemon, the client, compose, and even a local Kubernetes cluster in one single install."


So to summarize the architecture in one flow — you type a command into the Docker Client, it goes over the REST API to the Docker Daemon, the daemon delegates to containerd, containerd calls runc, runc talks to the Linux kernel using namespaces and cgroups, and your container is running. Images come from the Registry, data is stored in Volumes, and networking is handled by the bridge.

# Terminology of Docker and its Concepts

---

**1. Docker Image**

- A Docker Image is a read-only template used to create containers. I like to think of it exactly like a class in object-oriented programming — just like a class defines the blueprint, a Docker image defines everything the container needs to run — the OS, the runtime, the libraries, the application code.

- Images are built from a Dockerfile — a script of instructions. Each instruction in that Dockerfile adds a new layer to the image. These layers are cached, so if nothing changes, Docker reuses them — making builds faster.

- Once built, images are stored in a registry like Docker Hub — which is basically GitHub but for images. You push your image once and anyone on your team, or any server, can pull and run it.

---

**2. Docker Container?**

- A container is a running instance of a Docker image. If the image is the class, the container is the object created from that class.

- What makes containers special is that they are lightweight and isolated — each container gets its own filesystem, its own network, and its own process space. So two containers running on the same machine cannot interfere with each other.

- And just like you can create multiple objects from one class, you can run multiple containers from the same image — all isolated, all running independently. This is exactly how we scale applications — we just spin up more containers from the same image.

---

**3. Dockerfile**

- A Dockerfile is a plain text file that contains step-by-step instructions to build a Docker image. Docker reads this file from top to bottom and executes each instruction as a layer.

- For example — you start with FROM to pick a base image like Node.js 18. Then WORKDIR sets the working directory inside the container. COPY brings your code in. RUN executes commands like npm install to install dependencies. EXPOSE documents which port your app listens on. And CMD defines the default command that runs when the container starts.

- The beauty of a Dockerfile is that it makes your build 100% reproducible — anyone on your team runs the same Dockerfile and gets the exact same image, every single time.

---

**4. Docker Registry**

- A Docker Registry is a storage and distribution system for Docker images — think of it exactly like GitHub, but instead of storing code, it stores container images

- Docker Hub is the default public registry — it hosts millions of official images like Ubuntu, Nginx, Node, and Postgres that anyone can pull for free

- For private images in production, companies use private registries — AWS has ECR, Google has GCR, and Microsoft has ACR. These are secure, private registries integrated with their respective cloud platforms. So your proprietary application images never end up publicly accessible.

---

**5. Docker Engine**

- Docker Engine is the core runtime that powers everything in Docker — it is what actually builds images and runs containers on your machine. Without Docker Engine, nothing works

- It has three main parts working together — the Docker CLI which is what you type commands into, the REST API which carries those commands, and the Docker Daemon running in the background that actually processes everything — managing images, containers, networks, and volumes

- Internally, Docker Engine uses containerd to manage the container lifecycle and runc at the lowest level to actually create containers using Linux namespaces and cgroups. So when you type docker run, a whole chain of components fires up — but from your perspective, it just works."**

---

**6. Docker Daemon**

- Docker Daemon — written as dockerd — is the background service that does all the real work in Docker. It runs silently on the host machine and listens for Docker API requests.**

- Whenever you type any Docker command, the daemon is the one that actually executes it. It is responsible for managing all Docker objects — images, containers, networks, and volumes.**

- One more important thing — a daemon can also communicate with other daemons. This is how Docker manages services across multiple machines, like in Docker Swarm. So it is not just a single machine service — it can be part of a distributed system."**

---

**7. Docker Client**

- Docker Client — the docker command — is the primary way users interact with Docker. It is basically the Command Line Interface that we type all our commands into.**

- Here is how it works — when you type something like docker run, the client does not execute that itself. Instead it converts your command into a Docker API request and sends it to the Docker Daemon. The daemon receives it, carries it out, and sends the response back. The client then displays that result to you.**

- One thing that many people don't know is that the Docker Client can communicate with more than one daemon. So from your local machine, you can actually control and manage Docker on a completely different remote server — just by pointing the client to that daemon."**

---

**8. Docker Desktop**

- Docker Desktop is an easy-to-install application for Mac, Windows, and Linux that gives developers everything they need to build and run containers — all in one place.**

- What makes it powerful is what it bundles together. It includes the Docker Daemon, the Docker Client, Docker Compose for running multi-container applications, Docker Content Trust for image security, a built-in single-node Kubernetes cluster for local testing, and a Credential Helper for secure login management.**

- It also comes with a visual GUI dashboard — so you can manage your containers, images, and volumes without typing a single command. It is designed purely for developers on their laptops. In production on servers, you would use Docker Engine directly without Desktop."**

## How Docker Works Internally
---

Docker uses two core Linux kernel features:

**1. Namespaces (Isolation)**
   
Namespaces isolate container resources from the host and other containers:
- `pid` — Process isolation
- `net` — Network isolation
- `mnt` — Filesystem isolation
- `uts` — Hostname isolation

**2. cgroups (Resource Limiting)**
   
Control groups limit and monitor resource usage per container:
- CPU usage
- Memory limits
- Disk I/O

This is why containers are much **lighter than VMs** — there's no Guest OS overhead!
