# Docker Lifecycle 

## What is Docker Lifecycle?

- Docker Lifecycle is the **complete journey** of a container
- It starts from **writing a Dockerfile** and ends at **removing the container**
- Every container goes through **7 clear stages**

**1. Stage 1 — Write (Dockerfile)**

- Developer writes a **Dockerfile**
- Dockerfile contains **step-by-step instructions**:
  - Which base image to use (`FROM`)
  - Which dependencies to install (`RUN`)
  - Which files to copy (`COPY`)
  - Which port to open (`EXPOSE`)
  - Which command to run (`CMD`)
- This is the **starting point** of everything in Docker

---

**Stage 2 — Build (Docker Image)**

- Developer runs `docker build`
- Docker reads the Dockerfile **line by line**
- Each instruction creates a **new layer** in the image
- Layers are **cached** — if nothing changes, Docker reuses them
- Result is a **Docker Image** — read-only, portable, ready to use

---

**Stage 3 — Tag & Push (Registry)**

- Image is **tagged** with a version — like `v1.0` or `latest`
- Image is **pushed** to a Docker Registry:
  - Public → **Docker Hub**
  - Private → **AWS ECR / Google GCR / Azure ACR**
- Now the image is **stored, versioned, and shareable**
- Anyone on the team or any server can now access it

---

**Stage 4 — Pull (Download Image)**

- Any machine — dev laptop, test server, production — runs `docker pull`
- Docker **downloads the image** from the registry
- If the image is **already available locally**, Docker skips the download
- Image is now **ready to run** on that machine

---

**Stage 5 — Run (Container Created & Started)**

- Developer or server runs `docker run`
- Docker Engine creates a **container from the image**
- Internally:
  - **containerd** manages the container lifecycle
  - **runc** talks to the Linux kernel
  - **Namespaces** give the container isolation
  - **cgroups** limit CPU and memory usage
  - **OverlayFS** sets up the layered filesystem
- Container is now **live and running** 

---

**Stage 6 — Pause & Unpause (Optional)**

- `docker pause` → **freezes all processes** inside the container
- Uses Linux **cgroups freezer** — processes are suspended, not killed
- Container stays in memory — **nothing is lost**
- `docker unpause` → container **resumes instantly** from exact same state
- Useful when you need to **free up CPU temporarily**

---

**Stage 7 — Stop & Remove**

**Stop:**

- `docker stop` → sends **SIGTERM** first (graceful shutdown)
  - App gets time to finish requests, close DB connections, flush logs
  - After **10 seconds** — sends **SIGKILL** to force stop
- `docker kill` → sends **SIGKILL immediately** (no cleanup — emergency only)
- Container is now in **Stopped state** — it still exists, just not running

Important — Stopped ≠ Removed:
- A **stopped container can be restarted** with `docker start`
- It still exists on disk with its filesystem intact

**Remove:**

- `docker rm` → **permanently deletes** the container
- `docker rmi` → **permanently deletes** the image
- Any data inside the container's writable layer is **gone forever**


**What Happens to Data?**

- Data written **inside a container** lives in its writable layer
- When container is removed → that data is **lost forever**
- Solution → use **Docker Volumes**:
  - Volumes are stored **outside the container** on the host
  - Data **survives** even after container is deleted and recreated
  - Always use volumes in production for databases and important files

---

## Summary — 7 Stages in One Line Each

| Stage | Command | What Happens |
|-------|---------|--------------|
| **1. Write** | `vim Dockerfile` | Define how image is built |
| **2. Build** | `docker build` | Create image layer by layer |
| **3. Push** | `docker push` | Store image in registry |
| **4. Pull** | `docker pull` | Download image to machine |
| **5. Run** | `docker run` | Create and start container |
| **6. Pause** | `docker pause` | Freeze container processes |
| **7. Stop & Remove** | `docker stop` + `docker rm` | Shutdown and delete container |
