# Docker Compose

**Docker Compose**

- **Docker Compose** is a tool for defining and running **multi-container Docker applications**
- You describe your entire application stack in a single **YAML file** called `docker-compose.yml`
- With **one single command** — `docker compose up` — you start all your containers together
- Docker Compose handles:
  - Starting containers in the **right order**
  - Creating **shared networks** between containers
  - Managing **volumes** for persistent data
  - Setting **environment variables**
  - **Scaling** services up or down

- Think of Docker Compose as a **conductor of an orchestra** —
- each container is a musician, and Compose makes them all play together in harmony.

---

## Why Docker Compose?

**Without Docker Compose — Run Everything Manually**

```bash
# You have to run EACH container manually — one by one
docker network create myapp-network

docker run -d \
  --name postgres \
  --network myapp-network \
  -e POSTGRES_PASSWORD=secret \
  -v pgdata:/var/lib/postgresql/data \
  postgres:15

docker run -d \
  --name redis \
  --network myapp-network \
  redis:alpine

docker run -d \
  --name backend \
  --network myapp-network \
  -e DB_HOST=postgres \
  -e REDIS_HOST=redis \
  -p 5000:5000 \
  mybackend:latest

docker run -d \
  --name frontend \
  --network myapp-network \
  -p 80:80 \
  myfrontend:latest

# 4 commands just to start. What about stopping? Scaling? Updating?
```

**With Docker Compose — One Command Does Everything**

```bash
docker compose up -d
# 🎉 All 4 containers started, networked, and configured!
```

## 📄 Docker Compose File Structure

```yaml
version: '3.8'           # Compose file version

services:                # Define your containers here
  service-name:
    image: ...           # Which image to use
    build: ...           # Or build from Dockerfile
    ports: ...           # Port mapping
    volumes: ...         # Mount volumes
    environment: ...     # Environment variables
    depends_on: ...      # Start order dependency
    networks: ...        # Which network to join
    restart: ...         # Restart policy

volumes:                 # Define named volumes
  volume-name:

networks:                # Define custom networks
  network-name:
```

## Core Concepts

**1. Services**
- Each **service** is one container in your application
- A service has a name, an image or build context, and configuration
- Multiple instances of a service can run (scaling)

```yaml
services:
  web:          # ← service name (also DNS hostname)
    image: nginx
```

**2. Networks**
- Docker Compose **automatically creates a network** for your app
- All services on the same network can **talk to each other by service name**
- Service name becomes the **DNS hostname**

```yaml
# web service can reach db with hostname "db"
services:
  web:
    image: myapp
  db:
    image: postgres
# web connects to postgres at: postgresql://db:5432/mydb
```

**3.Volumes**
- Named volumes defined in compose file
- **Persist data** across container restarts and recreation
- Shared between services if needed

```yaml
services:
  db:
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:    # Docker manages this
```

**4.depends_on**
- Controls the **startup order** of services
- Makes sure dependencies start before the service that needs them

```yaml
services:
  backend:
    depends_on:
      - database    # database starts before backend
      - redis       # redis starts before backend
```

## Environment Variables

### Using a `.env` File

```bash
# .env file (never commit to Git!)
DB_PASSWORD=supersecret123
API_KEY=myapikey456
NODE_ENV=production
```

```yaml
# docker-compose.yml
services:
  backend:
    environment:
      - DB_PASSWORD=${DB_PASSWORD}    # reads from .env
      - API_KEY=${API_KEY}
      - NODE_ENV=${NODE_ENV}
```

### Using Multiple Env Files

```yaml
services:
  backend:
    env_file:
      - .env              # base variables
      - .env.production   # production overrides
```

### Override for Different Environments

```bash
# Development
docker compose -f docker-compose.yml -f docker-compose.dev.yml up

# Production
docker compose -f docker-compose.yml -f docker-compose.prod.yml up
```

## Docker Compose Networking

```yaml
version: '3.8'

services:
  frontend:
    networks:
      - public      # can be reached from outside

  backend:
    networks:
      - public      # frontend can reach it
      - private     # database can reach it

  database:
    networks
      - private     # ONLY backend can reach it — secure! 

networks:
  public:
    driver: bridge
  private:
    driver: bridge
```

```
Internet → frontend → backend → database
database is NEVER exposed to internet 
```

## Docker Compose Volumes

```yaml
version: '3.8'

services:
  database:
    image: postgres:15
    volumes:
      # Named volume — Docker manages location
      - pgdata:/var/lib/postgresql/data

      # Bind mount — you choose the path
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql

      # Read-only bind mount
      - ./config:/etc/postgresql:ro

  backend:
    volumes:
      # Share logs between host and container
      - ./logs:/app/logs

volumes:
  pgdata:      # persists after docker compose down
               # use -v flag to delete: docker compose down -v
```

## Summary


WHAT:   Tool to define and run multi-container apps
FILE:   docker-compose.yml (YAML format)
CMD:    docker compose up -d  → starts everything
        docker compose down   → stops everything

---

**WHAT IT MANAGES:**

  Services    → your containers
  Networks    → auto-created, DNS by service name
  Volumes     → persistent storage
  Order       → depends_on for startup sequence
  Config      → environment variables, .env files
  Scaling     → --scale backend=3

---

**KEY RULES:**

  Service name = DNS hostname between containers
  down keeps volumes, down -v deletes volumes
  depends_on = start order, not readiness
  Use .env file for secrets, never hardcode
  Use networks to isolate DB from internet
